## VULNERABLE Findings

### V1. Q5_K Advertised but Not Implemented ‚Äî Crash on Any Q5_K Model

**Files**: `quant.c:265-279`, `quant.c:519-534`, `quant.h:46`, `README.md:181,528`

**Seed**: README.md advertises "Q2_K, Q3_K, Q4_K, Q5_K, Q6_K, Q8_0, Q4_0, F16, F32" support. CONTRIBUTING.md lists Q5_K fused dot product as a contribution opportunity.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Q5_K was declared in the enum (`GGUF_TYPE_Q5_K = 13`) and in the type info tables (`gguf_type_block_size` returns 256, `gguf_type_quant_size` returns 176) but the actual dequantization kernel was never implemented.
- *"What does it actually do?"* ‚Äî `vec_dot()` falls through to the default case, which calls `dequantize_row()`, which also has no Q5_K case and calls `exit(1)`.
- *"What does this promise?"* ‚Äî README promises Q5_K support. Loading a Q5_K-quantized GGUF model will crash at the first matmul.

**Verification**: `quant.c:265-279` ‚Äî the `dequantize_row` switch has no `case GGUF_TYPE_Q5_K`, confirmed via grep. No `dequantize_row_q5_K` function exists anywhere in the codebase.

**Impact**: Any user downloading a Q5_K-quantized model (a common quantization on HuggingFace) will get a silent crash with `"dequantize_row: unsupported type 13"` and `exit(1)`. No graceful error at load time ‚Äî the crash happens during inference.

**WHY**: The developer added Q5_K to the enum and size tables as scaffolding (forward declaration), then documented it as supported before implementing it. The type-info functions work, creating the illusion that Q5_K is "known" to the system. But the critical path (dequantize/dot) was never completed. **Meta-pattern**: "Scaffolding creates false confidence of completeness." This is a declaration-vs-implementation gap.

**Radiation**: Checked Q4_1, Q5_0, Q5_1, Q8_1 ‚Äî same pattern. These are in the enum and type tables but have no `dequantize_row` case. However, only Q5_K is advertised in the README. The others are silently unsupported (enum-only), which is less misleading but still crashes if encountered.

**Verdict**: **VULNERABLE** ‚Äî confirmed crash path with false documentation.

### V2. GGUF Parser Has Zero Bounds Checking ‚Äî Out-of-Bounds Read on Malformed Files

**Files**: `model.c:45-94` (reader functions), `model.c:206` (parse_gguf entry)

**Seed**: The `reader_t` struct stores `size` but no read function ever checks `r->pos + N <= r->size`.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Assumed the model file is always well-formed because it came from a trusted source (HuggingFace, llama.cpp).
- *"What does it actually do?"* ‚Äî `read_u8`, `read_u16`, `read_u32`, `read_u64`, `read_f32`, `read_gguf_string` all access `r->data[r->pos]` without checking whether `r->pos` has exceeded `r->size`.
- *"What does this promise?"* ‚Äî The GGUF parser promises to parse model files. A truncated download (which the install script tries to catch but only checks if < 100MB), a corrupted file, or a crafted malicious file will cause out-of-bounds reads from the mmap'd region.

**Verification**: `model.c:42-43` defines `reader_t` with `size_t size` member. Grep for `r->size` or `r.size` in the parse path finds zero uses for bounds checking after `reader_t` initialization.

**Impact**: With `MAP_PRIVATE` and `PROT_READ`, out-of-bounds reads beyond the mmap region cause SIGSEGV/SIGBUS ‚Äî an opaque crash with no error message. A partially downloaded GGUF file (the most common real-world trigger) gives zero diagnostic information.

**WHY**: The developer stored `size` in `reader_t` (line 43), meaning they were *aware* bounds checking might be needed, but optimized for the happy path. **Meta-pattern**: "Storing the guard value without checking it creates the illusion of safety." The `size` field is defensive scaffolding that was never wired up.

**Radiation**: The same pattern appears in `tokenizer.c:8-13` (`read_u64_at`) ‚Äî no bounds checking on the pointer either.

**Verdict**: **VULNERABLE** ‚Äî confirmed missing validation with real-world trigger (truncated download).

### V3. Grammar Incorrectly Blocks Tokens Inside JSON Strings Containing Braces

**Files**: `grammar.c:12-41` (analyze_token), `grammar.c:90-112` (grammar_apply)

**Seed**: `analyze_token` pre-computes `token_brace_delta` by counting ALL `{`/`}` in a token's string, including those that would appear inside JSON string values. `grammar_apply` blocks tokens with negative brace delta (line 99) *before* checking `in_string` (line 105).

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Pre-computed per-token deltas for speed. The `in_string` skip at line 105 was meant to handle the string case, but it comes AFTER the negative-depth block.
- *"What does it actually do?"* ‚Äî When generating inside a JSON string (`g->in_string == 1`), tokens containing `}` have `token_brace_delta[i] < 0`. If `g->brace_depth + delta < 0`, the token is masked to `-inf` at line 100 BEFORE the `in_string` check at line 105 can skip it.
- *"What does this promise?"* ‚Äî "If we're in a string, don't constrain (allow any content)" (line 104 comment). But this promise is broken for tokens containing structural characters.

**Verification**: `grammar.c:98-105`:
```c
/* Prevent depth from going negative (unmatched closing) */
if (new_brace < 0 || new_bracket < 0) {
    logits[i] = NEG_INF;     // ‚Üê Blocks token BEFORE in_string check
    continue;
}

/* If we're in a string, don't constrain (allow any content) */
if (g->in_string) continue;  // ‚Üê Never reached for blocked tokens
```

**Impact**: When generating a JSON value like `{"code": "function() { return }"}`, the model cannot emit tokens containing `}` while inside the string if brace_depth is already at 1 (the outermost `{`). This distorts generation inside strings and could cause the model to avoid producing closing braces in string content, or get stuck if no alternative tokens exist.

**WHY**: The developer thought of the problem (the `in_string` skip exists) but placed it in the wrong order. The negative-depth check is a "safety net" that fires too eagerly. **Meta-pattern**: "Guard ordering determines correctness ‚Äî a correct check in the wrong position becomes a bug."

**Radiation**: The same issue affects bracket_depth and any token containing `]` inside strings.

**Verdict**: **VULNERABLE** ‚Äî confirmed logic ordering bug that corrupts JSON string content.

### V4. `tokenizer_decode` Returns Pointers to Static Buffers ‚Äî Not Thread-Safe, Not Re-entrant

**Files**: `tokenizer.c:231` (`static char byte_buf[2]`), `tokenizer.c:249` (`static char space_buf[256]`)

**Seed**: `tokenizer_decode` uses static buffers for byte-token and SentencePiece space handling.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Returns `const char*` without allocation, mimicking llama.cpp's pattern. Avoids malloc per token decode.
- *"What does it actually do?"* ‚Äî Each call overwrites the static buffer. Two consecutive calls clobber each other's results. In a multi-threaded context, data races occur.
- *"What does this promise?"* ‚Äî The returned pointer is valid "as a static/vocab string (do not free)" per the header comment.

**Verification**: `tokenizer.c:231`: `static char byte_buf[2]` is shared across all calls. If the caller saves the pointer from one `tokenizer_decode` call and then calls it again, the first pointer's data is overwritten.

**Impact**: Currently, `picolm.c` calls `tokenizer_decode` once per loop iteration and immediately `printf`s the result, so it works. But this is a latent fragility: any future code that stores decode results (e.g., for logging, concatenation, or server mode) will get corrupted data. The `CONTRIBUTING.md` lists "continuous batching for server mode" as a planned feature, which would hit this bug.

**WHY**: Single-threaded design assumption baked into the API contract. **Meta-pattern**: "Static buffers in output functions create invisible coupling between callers."

**Verdict**: **VULNERABLE** ‚Äî latent, but confirmed fragile API that will break on any multi-call or multi-thread usage. The planned server mode (CONTRIBUTING.md) would trigger this.

### V5. `token_has_unmatched_quote` Pre-Computed but Never Used

**Files**: `grammar.h:39`, `grammar.c:40,58,64-67`

**Seed**: `analyze_token` computes `has_unmatched_quote` for every token. `grammar_init` allocates and populates the array. But `grammar_apply` never reads it.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Intended to track string state transitions at the token level for faster masking.
- *"What does it actually do?"* ‚Äî The array is allocated, computed, stored, and freed ‚Äî consuming ~32KB of RAM for a 32K vocabulary ‚Äî but never consulted.
- *"What does this promise?"* ‚Äî The header comment says the grammar "tracks string state (inside vs outside quotes)." The per-token quote parity was clearly intended to power this, but it's dead code.

**Verification**: Grep for `token_has_unmatched_quote` finds: declaration in `grammar.h:39`, allocation in `grammar.c:58`, population in `grammar.c:64-67`, and free in `grammar.c:173`. Zero reads in `grammar_apply` or anywhere else.

**Impact**: The `in_string` tracking in `grammar_advance` (character-by-character) is the actual mechanism used. The pre-computed array is wasted work and memory. More importantly, its existence creates the false impression that string-awareness is deeply integrated into the grammar masking, when in reality `grammar_apply` only uses `g->in_string` (which is updated *after* sampling, not *during* masking).

**WHY**: The developer planned two-level string tracking: fast per-token parity for masking + accurate per-character for state. Only the second was wired up. **Meta-pattern**: "Dead infrastructure creates the illusion of feature completeness." Related to V1's scaffolding pattern.

**Verdict**: **VULNERABLE** ‚Äî confirmed dead code. 32KB wasted allocation + false documentation impression.

### V7. `sampler_sample` Allocates 375KB Per Token for Top-p Sampling

**Files**: `sampler.c:75-80`

**Seed**: Every call to `sampler_sample` with `top_p < 1.0` allocates `vocab_size * sizeof(prob_index_t)` via `malloc`.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Needed a sorted array for nucleus sampling. Used malloc because the array is large (~375KB for 32K vocab).
- *"What does it actually do?"* ‚Äî For each generated token: malloc 375KB, populate, qsort, sample, free. This is called once per generated token.
- *"What does this promise?"* ‚Äî `CONTRIBUTING.md:2` says "No malloc during inference. All memory is allocated at startup."

**Verification**: `sampler.c:75`: `prob_index_t *sorted = (prob_index_t *)malloc((size_t)vocab_size * sizeof(prob_index_t));`
`sizeof(prob_index_t)` = 4 (float) + 4 (int) = 8 bytes minimum, likely 8-12 with padding.
32000 * 8 = 256,000 bytes = 250KB minimum per token.
With 12 bytes (likely alignment): 32000 * 12 = 375KB per token.

This violates the project's own stated rule in CONTRIBUTING.md ("No malloc during inference").

**Impact**: On a Pi Zero with 512MB RAM, generating 256 tokens means 256 malloc/free cycles of 250-375KB each. Heap fragmentation on memory-constrained devices is a real concern. The per-token overhead of malloc + qsort(32K) + free adds measurable latency.

**WHY**: Top-p sampling was added later (it's optional ‚Äî temperature=0 doesn't trigger it). The "no malloc during inference" rule was established for the core forward pass but wasn't enforced for the sampling path. **Meta-pattern**: "Optional features bypass the design rules established for the core path."

**Verdict**: **VULNERABLE** ‚Äî confirmed malloc-per-token violating the project's own stated invariant.

### V8. `vec_dot` Fallback Allocates 32KB on Stack ‚Äî Stack Overflow Risk for Large Models

**Files**: `quant.c:526`

**Seed**: `float tmp[8192]` is a 32KB stack allocation inside `vec_dot`'s default path.

**Interrogation**:
- *"Why did you write it this way?"* ‚Äî Wanted to avoid malloc in the hot path. 8192 floats covers most model dimensions.
- *"What does it actually do?"* ‚Äî For non-Q4_K, non-Q6_K, non-F32 types, it allocates 32KB on the stack per `vec_dot` call. This function is called from `matmul_worker`, which runs on threads with default (or smaller) stack sizes.
- *"What does this promise?"* ‚Äî Falls back gracefully for unsupported-but-dequantizable types.

**Verification**: `quant.c:526`: `float tmp[8192];` = 32768 bytes on stack. `matmul` spawns threads via `pthread_create` with `NULL` attributes (default stack, usually 2-8MB on Linux, but configurable). Each thread calling `vec_dot` in the fallback path allocates 32KB of stack per row. With n_ffn=5632 (TinyLlama), that's `n / block_size` calls... no, actually only one call per row. But multiple threads each have their own 32KB stack allocation. If `n > 8192`, it falls back to `malloc`.

For TinyLlama, `n_embd=2048` and `n_ffn=5632`, both < 8192. For Llama 2 7B, `n_embd=4096` and `n_ffn=11008` > 8192, so it would malloc. The stack allocation is technically safe for TinyLlama but marginal for larger models, and the 32KB per thread is non-trivial on embedded systems where thread stacks may be reduced.

**WHY**: Same pattern as V7 ‚Äî "no malloc during inference" rule was the goal, but the fallback undermines it. **Meta-pattern**: "Fallback paths bypass the constraints of the optimized path."

**Verdict**: **VULNERABLE** ‚Äî confirmed marginal. 32KB stack allocation per thread in hot path, hitting malloc for 7B+ models despite the no-malloc design goal.

## HOLDS Findings

### H1. Prompt Token Buffer Allocation (`strlen(prompt) + 3`)

**File**: `picolm.c:168`

The assumption that BPE tokenization produces at most `strlen(prompt)` tokens is generally correct because BPE merges reduce token count. The `+3` provides margin for BOS. Even if truncation occurred, the tokens array is used with `max_tokens` bounds, so no overflow ‚Äî just silent prompt truncation. For practical inputs this is safe.

**Counter-evidence search**: Could a prompt produce more tokens than bytes? Only if single bytes expand to multi-byte token IDs (they don't ‚Äî the token ID is an int, not a string). After BPE merges, the count only goes down. The only expansion is the leading `‚ñÅ` added in normalization, but that becomes one token.

**Verdict**: **HOLDS** ‚Äî the allocation is sufficient for BPE tokenization.

### H2. `MAP_PRIVATE` vs `MAP_SHARED` for mmap

**File**: `model.c:176`

With `PROT_READ` (read-only), `MAP_PRIVATE` and `MAP_SHARED` behave identically ‚Äî the OS serves pages from the page cache without COW. COW only triggers on write, and `PROT_READ` prevents writes. No memory overhead.

**Verdict**: **HOLDS** ‚Äî `MAP_PRIVATE` is correct for read-only access.

### H3. `tokenizer_decode` Byte-Token Detection Hardcoded to Position 5

**File**: `tokenizer.c:229`

The check `str[0]=='<' && str[1]=='0' && str[2]=='x' && str[5]=='>'` reads `str[5]`. For tokens shorter than 6 characters that start with `<0x`, this reads past the end. However, the vocab tokens matching `<0x..>` are exactly 6 characters (e.g., `<0xFF>`), and other tokens starting with `<0x` are extremely unlikely in any real BPE vocabulary. The string is null-terminated, so `str[5]` reads the null byte or a valid character ‚Äî no memory violation, just a false-negative match.

**Verdict**: **HOLDS** ‚Äî safe for all real BPE vocabularies. The null terminator prevents UB.

### H4. xorshift64 RNG Quality

**File**: `sampler.c:9-16,18-21`

The `rand_float` function: `(xorshift64(state) >> 11) / (float)(1ULL << 53)`. This shifts out the bottom 11 bits of the 64-bit output, then divides by 2^53, producing a float in [0,1). xorshift64 passes basic statistical tests. The shift-right-by-11 ensures the top 53 bits (matching float64 mantissa width) are used. For LLM sampling, this quality is more than adequate.

**Verdict**: **HOLDS** ‚Äî appropriate RNG quality for the use case.

### H5. Flash Attention Accumulator Array Size

**File**: `model.c:644`

`float acc[256]` = 1KB on stack. `head_dim` is 64 for TinyLlama, 128 for Llama 2 7B. No known LLaMA-architecture model exceeds 256. However, there is no runtime assertion. This holds for all current models but has no defense against future ones.

**Verdict**: **HOLDS** ‚Äî safe for all known LLaMA models. Would benefit from a `assert(head_dim <= 256)` guard.

### H6. KV Cache Save/Load Validation

**Files**: `model.c:732-833`

`kvcache_save` validates `n_pos > 0 && n_pos <= seq_len`. `kvcache_load` validates magic, layer count, kv_dim match, and `n_pos <= seq_len`. On any mismatch, it returns 0 (no cache) and the caller proceeds without cache. This is robust.

**Verdict**: **HOLDS** ‚Äî proper validation with graceful fallback.

### H7. Online Softmax Numerical Stability

**File**: `model.c:659-672`

The implementation only computes `expf(x)` where `x <= 0` (either `old_max - new_max` when new max is found, or `score - max_score` otherwise). This prevents overflow. The initial `max_score = -1e30f` is safely below any real score. The algorithm is textbook online softmax.

**Verdict**: **HOLDS** ‚Äî numerically stable implementation.

### H8. Thread Safety of `matmul`

**File**: `tensor.c:54-121`

Each thread writes to a non-overlapping slice of the `out` array. Threads share read-only access to `x` and `W`. No shared mutable state. The barrier is `pthread_join`. This is correctly parallelized.

**Verdict**: **HOLDS** ‚Äî no data races in the matmul threading.

## UNCLEAR

### U1. Timing Measurement Off-by-One

**File**: `picolm.c:212-213`

`t_first_token` is set at `pos == n_prompt - 1`, which is the last prefill step. The first *generated* token isn't available until the next loop iteration. This means `prefill_time` includes the forward pass for the last prompt token but the first generated token's forward pass is counted in `gen_time`. Whether this is "correct" depends on convention. The first generated token's sampling happens in this same iteration, so the timing boundary is blurry.

**Verdict**: **UNCLEAR** ‚Äî subjective timing convention, not clearly wrong.

### U2. `install.sh` Appends to Shell Profile Without Idempotency

**File**: `install.sh:251-256`

The script checks `if ! grep -q "picolm/bin" "$PROFILE_FILE"` before appending, which is idempotent. However, the check uses `picolm/bin` as the grep pattern, which could false-match other paths. Low risk in practice.

**Verdict**: **UNCLEAR** ‚Äî minor, depends on user's environment.

### U3. `g_vocab_for_sort` Global State in Tokenizer

**File**: `tokenizer.c:17`

The global `g_vocab_for_sort` is set before `qsort` and used inside the comparator. This is a classic pattern forced by C's `qsort` API (no context parameter). It's not thread-safe, but `tokenizer_load` is only called once during initialization, never concurrently.

**Verdict**: **UNCLEAR** ‚Äî safe in current usage, but the global state is a code smell for any future multi-model loading.

## Root Causes (grouped by shared failed assumption + meta-pattern)

### Root Cause 1: "Scaffolding Creates False Confidence of Completeness"
**Affected**: V1 (Q5_K), V5 (token_has_unmatched_quote)
**Pattern**: Enum entries, struct fields, allocation code, and documentation are added as forward declarations, then never wired into the operational path. The existence of the scaffolding makes the feature *appear* complete in code review.

### Root Cause 2: "Optional/Fallback Paths Bypass Core Design Rules"
**Affected**: V7 (sampler malloc), V8 (vec_dot stack alloc)
**Pattern**: The project's "no malloc during inference" rule is enforced for the primary forward pass but violated by sampling (optional feature) and quantization fallback (edge case). Design rules erode at the boundaries.

### Root Cause 3: "Guard Ordering Determines Correctness"
**Affected**: V3 (grammar in_string)
**Pattern**: A correct check (`in_string ‚Üí skip constraints`) placed after an incorrect check (`negative depth ‚Üí block`) causes the correct check to never execute for the blocked tokens. The intent was right, the order was wrong.

### Root Cause 4: "Storing the Guard Value Without Checking It"
**Affected**: V2 (reader_t.size never checked)
**Pattern**: The developer was aware of the need for bounds checking (stored `size` in the struct) but never wired the check into the read functions. The field's existence creates a false sense of safety.
