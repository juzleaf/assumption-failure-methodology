## Root Causes (Grouped by Failed Assumption + Meta-Pattern)

### Root Cause 1: "All authenticated users are administrators"

The system sends the server-side `ENCRYPTION_KEY` to the frontend, exposes all passkey credentials without user filtering, and has no role-based access control. The fundamental assumption is that authentication = full administrative trust.

**Failed seeds**: #2, #3, #4 (ENCRYPTION_KEY leak), #20 (cross-user passkey access), #11 (emergency verification theater)

**Meta-pattern**: Admin-only data exposed through user-accessible API endpoints because the system conflates authentication with authorization.

### Root Cause 2: "Security checks are applied uniformly"

Validation logic, security constants, and safety checks are maintained in separate locations and have drifted apart. The export password minimum was silently weakened by a local variable shadow. The health check doesn't catch all OAUTH_ALLOW_ALL bypass values. Rate limiting fails open on DB errors.

**Failed seeds**: #7 (ALLOW_ALL='2' bypass), #13 (JWT expiry mismatch), #14 (rate limit fail-open), #17 (export password shadow), #22 (filename validation inconsistency)

**Meta-pattern**: Security configuration constants defined but not connected to the code that should use them, or applied in some paths but not all.

### Root Cause 3: "External data is safe to embed in HTML"

OAuth callback error responses from Google/Microsoft/Baidu/Dropbox are interpolated directly into inline `<script>` tags via template literals. Combined with CSP allowing `unsafe-inline` and `unsafe-eval`, this enables XSS.

**Failed seeds**: #5 (CSP unsafe-inline/unsafe-eval), #9 (postMessage wildcard), #10 (template literal HTML injection)

**Meta-pattern**: External API responses treated as trusted text and injected into HTML without escaping. CSP relaxed for framework compatibility eliminates the safety net.

### Root Cause 4: "Deployment environment provides trusted headers"

The system trusts `X-Forwarded-*` headers for WebAuthn origin determination and `CF-Connecting-IP` for rate limiting. These assumptions hold on Cloudflare Workers but break in Docker direct-exposure scenarios.

**Failed seeds**: #12 (WebAuthn rpID spoofing), #19 (shared rate-limit bucket in Docker)

**Meta-pattern**: Platform-specific header assumptions don't hold across deployment targets.

## Impact Chains

### Chain 1: Full Vault Compromise via Authenticated User (CRITICAL)

1. Any whitelisted user logs in via OAuth or Passkey
2. Login response / `/api/oauth/me` response contains `ENCRYPTION_KEY` in plaintext JSON (authService.ts:79, authRoutes.ts:143)
3. This key is the master key used to encrypt ALL 2FA secrets in the database (vaultService.ts:21)
4. If the database is ever leaked (backup exposure, SQL dump, etc.), the attacker can decrypt everything
5. Even without DB access, the same user already gets decrypted secrets via `/api/vault/` -- but the ENCRYPTION_KEY leaking means they can also decrypt any raw DB dump or backup file

**Evidence**:
- `backend/src/features/auth/authService.ts:79`: `...(needsEmergency && { encryptionKey: this.env.ENCRYPTION_KEY })`
- `backend/src/features/auth/authRoutes.ts:143`: `const encryptionKey = !isEmergencyConfirmed ? c.env.ENCRYPTION_KEY : undefined;`
- `backend/src/features/auth/webAuthnService.ts:173`: `...(needsEmergency && { encryptionKey: this.env.ENCRYPTION_KEY })`

### Chain 2: XSS via OAuth Callback Error Injection (HIGH)

1. Attacker initiates backup OAuth flow (e.g., Google Drive)
2. Attacker intercepts or manipulates the OAuth token exchange response
3. The `error_description` field contains `</script><script>malicious_code</script>`
4. backupRoutes.ts:106 interpolates this directly into an inline `<script>` in the HTML response
5. CSP allows `unsafe-inline` + `unsafe-eval` (config.ts:20-23), so the injected script executes
6. The script has access to the page context, including cookies and the refresh token being transmitted

**Evidence**:
- `backend/src/features/backup/backupRoutes.ts:106`: `` `Token exchange failed: ${errData.error_description || errData.error}` `` inside `<script>` tag
- Same pattern at lines 266, 410, 558 for Microsoft, Baidu, Dropbox callbacks
- `backend/src/app/config.ts:20-23`: CSP scriptSrc includes `'unsafe-inline'`, `'unsafe-eval'`

### Chain 3: Brute Force via Rate Limit Failure (MEDIUM)

1. Rate limiting depends on database operations
2. If the `rate_limits` table is missing/corrupted, or DB connection fails, errors are silently caught (rateLimitMiddleware.ts:104-108)
3. The request proceeds without rate limiting
4. Attacker can perform unlimited login attempts, WebAuthn attempts, or export requests
5. In Docker deployment, all users share rate-limit key `rl:unknown:...`, so one attacker's lockout affects all users

**Evidence**:
- `backend/src/shared/middleware/rateLimitMiddleware.ts:104-108`: `catch(e) { if (e instanceof AppError && e.statusCode === 429) throw e; console.error(...) }` -- all other errors pass through
- `backend/src/features/auth/authRoutes.ts:105`: `c.req.header('CF-Connecting-IP') || 'unknown'`

### Chain 4: Cross-User Passkey Manipulation (MEDIUM)

1. User A registers a passkey
2. User B logs in via OAuth
3. User B calls GET `/api/auth/webauthn/credentials` -- receives ALL passkeys including User A's
4. User B calls DELETE `/api/auth/webauthn/credentials/:id` with User A's credential ID
5. User A's passkey is deleted without authorization check

**Evidence**:
- `backend/src/features/auth/webAuthnService.ts:201-210`: `listCredentials()` has no `.where()` clause filtering by user
- `backend/src/features/auth/webAuthnService.ts:226-230`: `deleteCredential()` deletes by credential ID without checking ownership
- `backend/src/features/auth/webAuthnService.ts:216-220`: `updateCredentialName()` same issue

### Chain 5: Whitelist Bypass via Undocumented ALLOW_ALL Value (MEDIUM)

1. Administrator sets `OAUTH_ALLOW_ALL=2` (perhaps from outdated docs or confusion)
2. Health check at `health.ts:60` only flags `'true'` and `'1'` -- does NOT flag `'2'`
3. All API requests pass the security shield (no health check failure)
4. `authService.ts:91` accepts `'2'` as a valid ALLOW_ALL value, bypassing whitelist
5. ANY user from the configured OAuth provider can log in and access the vault

**Evidence**:
- `backend/src/features/auth/authService.ts:91`: `if (allowAllStr === 'true' || allowAllStr === '1' || allowAllStr === '2')`
- `backend/src/shared/utils/health.ts:60`: `if (String(env.OAUTH_ALLOW_ALL).toLowerCase() === 'true' || env.OAUTH_ALLOW_ALL === '1')`

## Defense Map (HOLDS + Coverage Gaps)

### CSRF Protection: HOLDS
- Double-submit cookie pattern correctly implemented
- CSRF token checked on all authenticated routes via `authMiddleware`
- Cookie attributes: `httpOnly: false` (readable by JS for header), `secure: true`, `sameSite: Lax`
- **Gap**: CORS reflects any origin, but CSRF header requirement prevents cross-origin exploitation

### OAuth State Validation: HOLDS
- State stored in httpOnly cookie, compared with client-provided state
- State cookie is one-time use (deleted after validation)
- **Gap**: None identified for standard OAuth flows

### SQL Injection Prevention: HOLDS
- Drizzle ORM parameterizes all queries
- LIKE search with user input: parameterized by ORM (no injection)
- **Gap**: LIKE wildcards (`%`, `_`) not sanitized, but this is UX, not security

### Input Sanitization: HOLDS (partial)
- `sanitizeInput()` removes HTML chars, control chars, invisible chars
- Base32 secret validation enforced on create/import
- **Gap**: Filename validation only on download/delete, not upload (SEED 22)

### Health Check Security Shield: HOLDS (partial)
- Blocks all API requests (except /health and /logout) when critical config is missing
- Checks ENCRYPTION_KEY length >= 32, JWT_SECRET length >= 32
- **Gap**: Does not detect `OAUTH_ALLOW_ALL=2` bypass (SEED 7)
