## Phase 2: Interrogation

### S01: Hardcoded P_COMM_OFFSET = 243 for macOS kinfo_proc

**"Why did you write it this way?"**
The developer needed to extract the process name (p_comm) from kinfo_proc, which is an opaque system struct. Rather than importing macOS C headers (which Zig's cross-compilation makes difficult), they hardcoded the struct offset. The value 243 is the known offset on both arm64 and x86_64 macOS for the p_comm field within extern_proc within kinfo_proc.

**"What does it actually do?"**
Calls sysctl to fill a raw byte buffer, then reads 16 bytes at offset 243 as a null-terminated process name.

**"What does this promise?"**
That offset 243 is stable across macOS versions for the p_comm field.

**REVERSE**: "What assumption, if false, would break this?"
If Apple changes the kinfo_proc struct layout (adds fields, reorders, changes alignment), offset 243 becomes wrong. This could read garbage data as a process name, or read past the valid buffer region.

Counter-evidence: Apple has kept kinfo_proc stable for decades due to ABI compatibility with existing tools (ps, top, lsof all rely on it). The struct is defined in `<sys/sysctl.h>` and is part of the stable KPI. However, ARM64 transition did NOT change the offset (verified by the developer's comment "both arm64 and x86_64"). Future architecture transitions or major macOS versions COULD change it, but historically this hasn't happened.

**FORWARD**: If the offset is correct, the 720-byte buffer must be large enough, and `size >= P_COMM_OFFSET + MAXCOMLEN` must hold. Both are validated by the bounds check on line 44.

**Verdict**: **HOLDS** (with caveat)

Challenge: Is offset 243 correct for ALL macOS versions currently in use (12-16+)?
Search: The kinfo_proc definition in xnu source shows p_comm at the same offset consistently. The code has an explicit bounds check (`if (size < P_COMM_OFFSET + MAXCOMLEN) return null`), so even if the offset is wrong on some future version, it fails safely to null rather than reading garbage. This is good defensive coding.

**Confirmed HOLDS**: The offset is historically stable, and the bounds check prevents unsafe reads even if it ever changes.

### S02: info_buf size 720 for kinfo_proc (~648 bytes)

**"Why did you write it this way?"**
The developer knew kinfo_proc is ~648 bytes on arm64 macOS and added ~72 bytes of headroom for potential future growth or alignment differences.

**"What does it actually do?"**
Allocates a stack buffer of 720 bytes with 8-byte alignment, passes its address and length to sysctl.

**"What does this promise?"**
That 720 bytes is sufficient for kinfo_proc on any supported macOS + arch combination.

**REVERSE**: If kinfo_proc grows beyond 720 bytes, sysctl will fail with ENOMEM (or return partial data depending on the syscall behavior). The code checks `if (size == 0) return null` but does NOT check the return value for ENOMEM-style failures — it checks `!= 0` which catches this.

**Verdict**: **HOLDS**

Challenge: What if kinfo_proc is larger on some macOS version?
The sysctl call returns error if buffer is too small, and the code checks for `!= 0` return, so this fails safely.

### S03: macOS getPtyName returns 0 unconditionally

**"Why did you write it this way?"**
macOS doesn't have `ptsname_r` (the thread-safe version). The developer chose to simply not report the PTY name on macOS rather than using the non-thread-safe `ptsname()` or doing a `ttyname_r` call (which is used in `openPtyUnix` but not here).

**"What does it actually do?"**
Returns 0, meaning `ForkResult.pty_name_len` is 0. In the NAPI layer (`pty_unix.zig:146-149`), if `name.len == 0`, the "pty" property is simply not set on the JS result object.

**"What does this promise?"**
That PTY name is optional and callers don't depend on it.

**REVERSE**: If any code path requires the PTY name to be present on macOS, it will get `undefined` and potentially crash. Let me check...

In `src/pty/unix.ts:50`, `this._pty = result.pty` — but on macOS, `result.pty` would be `undefined`. Then `get process()` at line 80-86 calls `native.process(this._fd)` which is a separate mechanism. So `_pty` is stored but never used for anything critical.

In `src/spawn.test.ts:188`, the open test checks `result.pty` matches a regex — but that's `open()`, which uses `ttyname_r` on the slave fd, not `fork()`.

**FORWARD**: Since fork on macOS doesn't set pty name, any consumer expecting `result.pty` to be a string will get undefined. The TS type says `pty: string` (line 17 of napi.ts) — this is a type lie on macOS.

**Verdict**: **VULNERABLE**

**Evidence**: `INativeUnix.fork()` return type declares `pty: string` but on macOS it returns `undefined` (property not set). Any code doing `result.pty.length` or `result.pty.startsWith(...)` will throw TypeError on macOS.

**WHY**: The developer assumed the pty name was a nice-to-have that nobody would access directly. The TypeScript type signature was written to match the Linux behavior and wasn't adjusted for macOS's missing implementation. The false confidence came from the fact that the internal code never uses `_pty` after storing it.

**Meta-pattern**: **Type signature diverges from runtime behavior across platforms** — the same interface promises different things on different OSes.

### S04: macOS TIOCSWINSZ hardcoded as 0x80087467

**"Why did you write it this way?"**
Zig's standard library has TIOCSWINSZ for Linux (via `std.posix.T.IOCSWINSZ`) but the macOS constant isn't readily available in Zig's stdlib. The developer hardcoded the known value.

**"What does it actually do?"**
Passes this ioctl number when resizing the PTY on macOS.

**"What does this promise?"**
That `0x80087467` is the correct TIOCSWINSZ value on all macOS versions.

**REVERSE**: If this value changes, resize would fail silently (ioctl returns error, which IS checked and returns `IoctlFailed`). But TIOCSWINSZ is defined in `<sys/ioctl.h>` as `_IOW('t', 103, struct winsize)`, and the encoding is architecture-independent on macOS.

**Verdict**: **HOLDS**

Challenge: Is this the correct encoding?
`_IOW('t', 103, struct winsize)` = 0x80000000 | (sizeof(winsize)<<16) | ('t'<<8) | 103 = 0x80000000 | (8<<16) | (0x74<<8) | 0x67 = 0x80087467. Confirmed correct. The macro is stable since it's derived from sizeof(winsize) which is 8 bytes (fixed struct).

### S05: forkPtyUnix — arena allocator lifetime across fork

**"Why did you write it this way?"**
The NAPI layer (`pty_unix.zig:49-51`) allocates an ArenaAllocator for all string conversions, defers its cleanup. The pointers from this arena are passed via `ForkOptions` to `forkPtyUnix`, which passes them to `forkpty()` and then the child calls `execChild()`.

**"What does it actually do?"**
After `forkpty()`, the child process has a copy-on-write clone of the parent's memory. The child calls `execChild()` which calls `execvpe`/`execvp` with the arena-allocated pointers. Since fork creates a full address space copy, the pointers are valid in the child until exec replaces the address space.

**"What does this promise?"**
That the arena memory remains valid in the child between fork and exec.

**REVERSE**: In the parent process, after `forkpty()` returns (pid > 0), the function continues to use `master_fd` and eventually returns. The arena is deferred to be freed when `forkImpl` returns. By that time, the child has either already exec'd (replacing its address space, so the arena memory no longer matters) or is still running pre-exec code. The parent freeing the arena does NOT affect the child because fork creates independent address spaces (COW).

**Verdict**: **HOLDS**

Challenge: What if the child hasn't exec'd yet when the parent frees the arena?
Fork creates a separate address space. Parent's free doesn't affect child memory. This is fundamental Unix semantics.

### S06: Signal mask manipulation around forkpty

**"Why did you write it this way?"**
Blocking all signals before forkpty prevents signal handlers from running during the critical fork window, which could corrupt state. This is standard practice from `posix_spawn` implementations.

**"What does it actually do?"**
1. Fills sigset to block everything
2. Sets the mask (saving old mask)
3. Calls forkpty
4. In both parent and child, restores the old mask
5. Child also calls `resetSignalHandlers()` to set all to SIG_DFL

**"What does this promise?"**
That no signal can interfere between fork and the child's signal reset.

**REVERSE**: If forkpty internally installs signal handlers... forkpty is just fork + openpty + setsid + ioctl. It doesn't install signal handlers. The signal mask is properly restored in both processes.

**Verdict**: **HOLDS**

Challenge: Is there a window where the mask isn't restored?
If forkpty returns -1 (error), the mask is restored on line 121. If child exec fails, `process.exit(1)` is called which terminates the child (mask doesn't matter for a dying process). All paths restore the mask in the parent.

### S07: closeExcessFds called after setuid/setgid

**"Why did you write it this way?"**
The ordering in `forkPtyUnix` is: chdir → setgid → setuid → closeExcessFds → execChild. The close happens after privilege drop.

**"What does it actually do?"**
Closes all file descriptors >= 3 in the child process. On macOS, this iterates from 3 to sysconf(_SC_OPEN_MAX). On Linux, uses close_range or /proc/self/fd.

**"What does this promise?"**
That closing FDs after dropping privileges still works.

**REVERSE**: File descriptors don't have per-user permissions in the traditional Unix model. `close()` succeeds on any open FD regardless of the current uid. The concern would be if the FD itself was somehow protected, but Unix FDs are process-local resources. So setuid before closeExcessFds is fine.

However, there's a subtler issue: the ORDER should arguably be closeExcessFds FIRST, then setuid/setgid. Reason: if there are high-privilege FDs open (e.g., FD pointing to /etc/shadow), you want to close them BEFORE dropping to the target user. The current order leaves a brief window where the child process runs as the target uid but still has the parent's open FDs. But this window is between setuid and closeExcessFds — a few microseconds — and the child hasn't exec'd any user code yet, so there's no code that could exploit these FDs.

**Verdict**: **HOLDS** (with nuance)

The ordering is slightly suboptimal (best practice is close-before-setuid) but not exploitable because no user code runs in the window between setuid and close.

### S08: Child chdir failure — no diagnostic to parent

**"Why did you write it this way?"**
After fork, the child process has a very limited communication channel back to the parent. The standard pattern is to exit with a non-zero code. The code exits with code 1 on chdir failure.

**"What does it actually do?"**
Parent sees the child exit with code 1 via waitpid in the exit monitor thread.

**"What does this promise?"**
That exit code 1 adequately communicates "something went wrong in setup."

**REVERSE**: Exit code 1 is ambiguous — it could mean chdir failed, setuid failed, setgid failed, or exec failed. The parent cannot distinguish these failure modes. This is a common limitation of the fork-exec model.

A more robust approach would be the "error pipe" pattern: create a CLOEXEC pipe before fork, write the error in the child, read it in the parent. But this adds complexity.

**Verdict**: **HOLDS** (design tradeoff, not a bug)

This is a known limitation accepted by most PTY libraries (including node-pty). The developer chose simplicity over diagnostic richness.

### S09: setuid/setgid without initgroups/setgroups

**"Why did you write it this way?"**
The developer implemented basic privilege setting: set GID then UID. This is the standard order (must set GID before UID because setting UID may drop privilege to call setgid).

**"What does it actually do?"**
Calls `setgid(gid)` and `setuid(uid)` if the options specify them.

**"What does this promise?"**
That the child runs as the specified uid:gid.

**REVERSE**: Without `initgroups()` or `setgroups()`, the child retains the PARENT process's supplementary groups. If Node.js is running as root with supplementary groups (wheel, sudo, etc.), and the child should run as user "nobody", the child will still be in those supplementary groups. This could grant unintended file access.

This is a real security concern for privilege escalation scenarios. However, the uid/gid options default to null (-1 from JS), meaning they're rarely used in practice. Most PTY usage doesn't involve uid/gid switching.

**Verdict**: **VULNERABLE**

**Evidence**: When uid/gid options are used to drop privileges, the child inherits parent's supplementary groups. A child running as uid=1000 but retaining group "wheel" could access files with group-wheel permissions.

**WHY**: The developer implemented the common pattern (setgid-then-setuid) but missed the supplementary groups step. This is a frequently overlooked aspect of privilege dropping — most tutorials and examples only show setuid/setgid without mentioning initgroups. The false confidence came from getting the uid/gid ordering right (which is the more commonly known gotcha).

**Meta-pattern**: **Incomplete privilege-dropping sequence** — getting the visible parts right (ordering) while missing the invisible parts (supplementary groups).

**"Why did you write it this way?"**
`openPty()` opens a bare PTY pair without forking a process. The developer likely reasoned that the caller gets raw master/slave FDs and can configure termios themselves.

Calls `unix.openpty()` with null termios pointer (line 167: `null, @ptrCast(&ws)`) — the PTY gets kernel defaults.

That the caller of the bare `open()` API handles their own termios configuration.

Actually, wait — the kernel default for a NEW PTY on most systems IS with reasonable defaults (echo on, canonical mode). Let me reconsider. The default termios for a newly allocated PTY is system-dependent, but generally includes ECHO and ICANON. The `forkPty` path explicitly sets termios because it wants specific behavior, but the defaults may already be acceptable for many use cases.

### S11: waitForExitUnix blocks on waitpid — zombie race

**"Why did you write it this way?"**
The exit monitor thread calls `waitpid(pid, &status, 0)` in a loop, which blocks until the child changes state.

**"What does it actually do?"**
Blocks until the child process exits (or is signaled). Handles EINTR by retrying.

**"What does this promise?"**
That waitpid will eventually return the child's exit status.

**REVERSE**: "What if the child exits before the thread starts?" — This is the zombie race question. Answer: In Unix, when a child exits before the parent calls waitpid, the child becomes a zombie (it stays in the process table with its exit status preserved). The subsequent waitpid call will immediately return with the zombie's status. This is exactly what zombies are for.

**Verdict**: **HOLDS**

Challenge: What if another thread calls waitpid on the same pid?
There's only one exit monitor thread per child. Node.js doesn't waitpid on arbitrary pids. So no double-waitpid issue.

### S12: ExitInfo heap allocation leak on tsfn call failure

**"Why did you write it this way?"**
In `exitMonitorThread` (pty_unix.zig:22-31), `waitForExit` returns info on the stack, then the code allocates ExitInfo on the heap and passes it to the threadsafe function. The JS callback (`exitCallJs` in pty.zig:37-38) does `defer alloc.destroy(data)`.

**"What does it actually do?"**
If `napi_call_threadsafe_function` fails (returns non-ok), the function continues to the defer block which releases the tsfn and destroys the context. But `exit_data` was allocated on line 24 and is only freed in `exitCallJs`. If the tsfn call fails, exitCallJs never runs, and exit_data leaks.

**"What does this promise?"**
That the tsfn call always succeeds or that the leak is acceptable.

**REVERSE**: The tsfn call can fail if the function was already released/aborted (e.g., the JS side called close). In this case, the `alloc.create(lib.ExitInfo)` on line 24 leaks.

**Verdict**: **VULNERABLE**

**Evidence**: If `napi_call_threadsafe_function` returns non-ok at line 27, `exit_data` (allocated on line 24) is never freed. The function returns via defer which only handles the tsfn and ctx cleanup, not exit_data.

**WHY**: The developer followed the pattern of "allocate, pass ownership to callback." But didn't handle the case where the callback never fires. The false confidence came from the fact that the callback has a `defer alloc.destroy(data)` — which looks like proper cleanup but only works if the callback actually runs.

**Meta-pattern**: **Ownership transfer failure** — allocating memory to be freed by a callback that may never execute.

## Root Causes Grouped by Shared Failed Assumption

### RC1: "Cleanup that runs twice is better than cleanup that runs zero times"
**Shared by**: S32, S42
**Failed assumption**: Calling close() twice on a resource is harmless if wrapped in try-catch.
**Reality**: File descriptor double-close is actively dangerous because the fd number can be reused between the first and second close, causing the second close to close an unrelated file descriptor. Try-catch hides the error but doesn't prevent the damage.

### RC2: "Callbacks always fire if I queue them"
**Shared by**: S12, S27
**Failed assumption**: Memory allocated for a callback's consumption will always be freed by that callback.
**Reality**: If the callback mechanism fails (tsfn aborted, JS thread shut down), the callback never runs and the allocated memory leaks. On the flip side, the exit monitor may free context before the callback fires (S27), creating use-after-free.

### RC3: "First output means ready"
**Shared by**: S35
**Failed assumption**: ConPTY processes always produce stdout before needing stdin.
**Reality**: Input-first processes (cat, custom tools) produce no output until they receive input, creating a deadlock with deferred writes.

### RC4: "Type signatures match runtime behavior"
**Shared by**: S03
**Failed assumption**: The TypeScript interface accurately describes what the native layer returns on all platforms.
**Reality**: On macOS, `fork()` doesn't set the `pty` property, but the TypeScript type declares it as `string`.

### RC5: "String equality is sufficient for control character filtering"
**Shared by**: S33
**Failed assumption**: Flow control characters arrive as isolated single-character data chunks.
**Reality**: Terminal data arrives in chunks of variable size; control characters can be embedded in larger chunks.

### RC6: "Exit cleans everything up"
**Shared by**: S39, S41
**Failed assumption**: Clearing listener arrays on exit is sufficient cleanup.
**Reality**: Pending async operations (waitFor timers, WriteQueue writes) continue running after listener arrays are cleared, producing confusing behavior or operating on stale resources.

### RC7: "Native blocking calls are fast enough for the JS thread"
**Shared by**: S21
**Failed assumption**: Writing to a ConPTY pipe returns quickly.
**Reality**: If the child process doesn't consume stdin, the pipe fills up and WriteFile blocks indefinitely, freezing the Node.js event loop.

### RC8: "Security features only need the obvious parts"
**Shared by**: S09
**Failed assumption**: setgid + setuid is complete privilege dropping.
**Reality**: Supplementary groups are a separate dimension of privilege that persists across setuid/setgid calls. `initgroups()` or `setgroups()` is required for complete privilege dropping.

### RC9: "page_allocator is fine for small allocations"
**Shared by**: S23
**Failed assumption**: The allocator choice doesn't significantly impact memory for small structures.
**Reality**: page_allocator allocates minimum 4KB per allocation. A 16-byte DataChunk struct wastes 99.6% of its allocation.

## Impact Chains

### Chain 1: FD Double-Close → Silent Corruption
```
UnixPty.close() → _readable.destroy() closes fd N → fd N recycled by Node → fs.closeSync(N) closes NEW resource → corrupted I/O on unrelated resource
```
**Impact**: Silent data corruption or dropped connections in the Node.js process.
**Trigger**: Any call to `close()` on a UnixPty or standalone Terminal.

### Chain 2: Windows Write Deadlock
```
User calls pty.write(largeData) → native writeInput() blocks on full pipe → JS event loop frozen → tsfn callbacks can't process → exit events delayed → entire Node.js app unresponsive
```
**Impact**: Application freeze (denial of service to self).
**Trigger**: Writing to a Windows ConPTY where the child process doesn't consume stdin.

### Chain 3: Use-After-Free on Windows Exit
```
Process exits → exit monitor: destroy(ctx) → JS event loop: processes pending write/resize call → native: dereferences freed ctx pointer → undefined behavior (crash or corruption)
```
**Impact**: Memory corruption, crash.
**Trigger**: Calling write/resize after process exits but before JS exit callback fires (race window).

### Chain 4: ConPTY Readiness Deadlock
```
spawn(cat) → no stdout → _ready stays false → pty.write() deferred → cat waits for stdin → deadlock
```
**Impact**: Process and PTY permanently blocked.
**Trigger**: Spawning an input-first process on Windows.

### Chain 5: waitFor Zombie Timer
```
Process exits quickly without matching pattern → _handleExit clears _dataListeners → waitFor timer still ticking → 30 seconds later: timeout rejection → confusing error message
```
**Impact**: Poor UX, 30-second delay for what should be an instant error.
**Trigger**: `waitFor()` on a fast-exiting process.

## Defense Map (HOLDS + Coverage Gaps)

### Strengths
1. **Signal handling around fork** (S06): Textbook correct — block all, fork, restore, reset in child.
2. **ConPTY cleanup ordering** (S24, S25): Complex but carefully designed with atomic guards and proper thread joining.
3. **EAGAIN handling in WriteQueue** (S53): Properly handles non-blocking write retry with close-awareness.
4. **Arena allocator for fork strings** (S05, S14): Clean pattern — arena lives in parent, child has its own copy.
5. **Three-tier FD close on Linux** (S49): Robust fallback chain.
6. **HRESULT checking** (S18): Correct FAILED check pattern.
7. **Native loading fallback** (S44): glibc→musl fallback handles Linux distro diversity.
8. **Command line quoting** (S22): Correct implementation of Windows rules.
9. **Defensive kinfo_proc bounds check** (S01, S02): Even if offset changes, fails safely.

### Coverage Gaps
1. **No fd ownership tracking**: No mechanism to know whether ReadStream or explicit code "owns" the fd. This leads to double-close (S32, S42).
2. **No cross-thread ownership protocol**: Memory allocated in one thread and consumed in another has no formal ownership transfer (S12, S27).
3. **No async operation cancellation on exit**: Pending promises/timers aren't cancelled when the PTY exits (S39).
4. **No backpressure from native to JS on Windows writes**: Synchronous native write can block JS thread indefinitely (S21).
5. **No platform-conditional type narrowing**: TypeScript types don't reflect platform-specific differences (S03).
6. **No WriteQueue lifecycle management on reattach**: Terminal._attachUnixFd doesn't clean up the old WriteQueue (S41).
7. **No initgroups in privilege dropping**: Supplementary group handling is missing (S09).
