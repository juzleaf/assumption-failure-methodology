### PROVEN-1: macOS `environ` race condition — thread-unsafe global mutation before fork

**Assumption that failed**: Setting the global `environ` pointer before `execvp` in the child process is safe because fork has already happened.

**File**: `zig/pty_darwin.zig:12-14`
```zig
const environ: *[*:null]const ?[*:0]const u8 = @extern(*[*:null]const ?[*:0]const u8, .{ .name = "environ" });
environ.* = envp;
_ = execvp(file, argv);
```

**Concrete proof**: The `environ` global is written in the child process after `forkpty()` returns `pid==0`. However, `forkpty()` is called while signals are masked (`sigprocmask(SIG.SETMASK, &sigset_all, ...)`), but **`environ` is a process-global variable visible to all threads in the parent process**. On macOS, `fork()` copies the parent's address space, so the child gets its own copy — the write is safe in isolation. BUT: the real danger is that `lib.zig:118` calls `forkpty()` which internally calls `fork()`. Between the signal mask and the fork, another thread in the parent could be modifying `environ` via `setenv()`/`putenv()`. The child inherits whatever partial state `environ` was in at the moment of fork.

More critically: if two `spawn()` calls happen concurrently on macOS, the signal masking in `forkPtyUnix` (`lib.zig:113-115`) blocks signals but does NOT prevent another thread from calling `forkPtyUnix` concurrently. Two threads could race on:
1. Thread A: masks signals, calls `forkpty()`, child A sets `environ = envpA`, calls `execvp`
2. Thread B: masks signals, calls `forkpty()`, child B sets `environ = envpB`, calls `execvp`

The children are fine (separate address spaces after fork). But the parent's `forkpty()` calls are not serialized — `forkpty()` itself is not thread-safe on all platforms. On macOS, calling `forkpty()` from two threads simultaneously is undefined behavior per POSIX (`fork()` is async-signal-safe but not thread-safe with respect to other `fork()` calls in the presence of `pthread_atfork` handlers).

**Reproduction**: Call `spawn()` from two Node.js worker threads simultaneously on macOS. The `forkpty()` calls can interleave, and `pthread_atfork` handlers registered by libc/malloc can deadlock or corrupt state.

**Root cause / WHY**: The developers assumed Node.js is single-threaded, so concurrent `forkPty` calls cannot happen. But Node.js worker threads, libuv thread pool callbacks, and NAPI threadsafe function callbacks can all lead to concurrent calls. The signal masking protects against signal delivery but not against concurrent fork.

**Meta-pattern**: "Single-threaded assumption in a library consumed by multi-threaded runtimes."

### PROVEN-2: Flow control check uses strict equality on single characters — multi-byte chunks bypass it

**Assumption that failed**: Comparing `data === this._flowControlPause` catches XOFF characters in the PTY output stream.

**File**: `src/pty/unix.ts:69-71`
```typescript
if (this.handleFlowControl && typeof data === "string") {
    if (data === this._flowControlPause || data === this._flowControlResume) return;
}
```

**Concrete proof**: The check uses `===` (strict equality) comparing the entire `data` chunk against the single-character flow control string (`"\x13"` or `"\x11"`). PTY output arrives in arbitrary-sized chunks from `tty.ReadStream`. A chunk will almost never be exactly one byte containing only the XOFF/XON character.

Failing input construction:
- The PTY delivers `"hello\x13world"` as a single chunk (common — ReadStream buffers).
- `data === "\x13"` is `false` because data is `"hello\x13world"`, not `"\x13"`.
- The XOFF character passes through to all listeners unintercepted.
- Only when the kernel happens to deliver exactly one byte containing `\x13` (rare) does the filter trigger.

This means flow control interception is effectively broken for any realistic data stream. The correct implementation should use `data.includes()` or process character-by-character, and strip the flow control characters rather than dropping the entire chunk.

**Root cause / WHY**: Direct port from node-pty which has the same bug. The assumption was that flow control characters arrive as isolated single-character reads. In practice, `tty.ReadStream` coalesces reads, so single-character delivery is a race condition, not a guarantee.

**Meta-pattern**: "Exact-match filtering on variable-granularity stream chunks."

### PROVEN-3: `UnixPty.close()` sends SIGHUP without awaiting exit — `_handleExit` races with `close` state

**Assumption that failed**: `close()` safely terminates the process and cleans up resources without race conditions.

**File**: `src/pty/unix.ts:120-138`
```typescript
close(): void {
    if (this._closed) return;
    this._closed = true;
    this._wq.close();
    try { this._readable?.destroy(); } catch {}
    try { fs.closeSync(this._fd); } catch {}
    try {
        process.kill(this.pid, 0);
        process.kill(this.pid, "SIGHUP");
    } catch {}
}
```

**Concrete proof**: The sequence is:
1. `this._closed = true` — sets the closed flag
2. `fs.closeSync(this._fd)` — closes the master FD
3. `process.kill(this.pid, "SIGHUP")` — sends SIGHUP

But the exit monitor thread (`zig/pty_unix.zig:16-31`) is running `waitpid()` concurrently. When the process exits (from SIGHUP or from the FD close), `waitpid` returns and calls the JS `onExit` callback via `napi_threadsafe_function`. This callback invokes `_handleExit` (`src/pty/_base.ts:125-136`), which sets `this._closed = true` and clears `_dataListeners` and `_exitListeners`.

Race scenario:
1. User calls `close()` — sets `_closed = true`, closes FD, sends SIGHUP
2. Meanwhile, `_handleExit` fires from exit monitor thread (via tsfn on next tick)
3. `_handleExit` sets `_closed = true` again (harmless) but also calls `this._resolveExited(info.exitCode)` and clears listeners
4. This is benign in simple cases, BUT: `close()` does NOT resolve the `exited` promise. If the process was already killed and the exit callback was already queued but not yet delivered, `close()` closes the FD, which causes the `tty.ReadStream` to emit an error. The `_readable.on("error", () => {})` handler swallows it.
5. If the exit monitor's tsfn callback fires AFTER `close()` already destroyed the readable and closed the FD, `_handleExit` tries to destroy `_readable` again in the onExit callback (`src/pty/unix.ts:43-45`): `this._readable?.destroy()` — this is called on an already-destroyed stream via the closure captured at construction time.

The concrete failing sequence:
```typescript
const pty = spawn("/bin/sleep", ["60"]);
pty.close();  // closes FD, sends SIGHUP
// exit monitor fires _handleExit → tries this._readable?.destroy() on already-destroyed stream
// The exited promise resolves (good), but only after close() already returned
await pty.exited; // This works by accident because _handleExit still fires
```

The real bug: if `close()` is called and then the caller immediately discards the pty object, the `_handleExit` callback fires asynchronously and accesses a potentially garbage-collected object. Node.js prevents GC of objects with live closures, but the tsfn reference in Zig holds a raw pointer to a JS function — if the exit monitor thread fires the tsfn after the JS engine has cleaned up the closure scope, it's a use-after-free at the NAPI level.

More concretely: `close()` never calls `napi_release_threadsafe_function` on the exit tsfn. The exit monitor thread in Zig (`zig/pty_unix.zig:16-31`) owns the tsfn and releases it after `waitpid` returns. But if the JS side has fully GC'd the callback, the tsfn call fires into a dead callback. This is prevented by the tsfn reference count keeping the callback alive — but it means the exit monitor thread (and its memory) stays alive indefinitely if the child process doesn't exit promptly after `close()`.

**Root cause / WHY**: `close()` does not coordinate with the exit monitor thread. It relies on SIGHUP eventually killing the process, which then causes the exit monitor to clean up. But there's no guarantee SIGHUP kills the process (it can be caught/ignored).

**Meta-pattern**: "Fire-and-forget cleanup with no coordination of asynchronous teardown."

### PROVEN-4: `waitFor` leaks timer and listener if the PTY process exits before pattern is matched

**Assumption that failed**: `waitFor` always resolves or rejects — either the pattern matches or the timeout fires.

**File**: `src/pty/_base.ts:73-123`

**Concrete proof**: `waitFor` sets up:
1. A `setTimeout` timer (for timeout rejection)
2. Either a terminal `_dataListeners` entry or an `onData` disposable

If the process exits before the pattern is matched AND before the timeout fires:
- `_handleExit` (`_base.ts:125-136`) sets `_closed = true` and clears `_dataListeners` (line 130)
- The `waitFor`'s listener is removed from the array by `_dataListeners.length = 0`
- No more data will arrive, so the pattern can never match
- The Promise is now stuck: it will neither resolve (no pattern match) nor reject until the timeout fires

This means if you do:
```typescript
const pty = spawn("/bin/sh", ["-c", "exit 0"]);
try {
    await pty.waitFor("something-that-never-appears", { timeout: 30000 });
} catch (e) {
    // Won't be reached for 30 seconds
}
```

The process exits immediately, but `waitFor` hangs for the full 30 seconds until the timeout fires. The timer reference keeps the Node.js event loop alive, preventing the process from exiting cleanly.

There is no `onExit` listener in `waitFor` to detect process exit and reject early. The cleanup function is only called by the timer or by pattern match — never by process exit.

**Root cause / WHY**: `waitFor` was designed assuming the pattern will eventually appear or the timeout will fire. Process exit as a third termination condition was not considered.

**Meta-pattern**: "Promise that depends on data from a source that can terminate independently."

### PROVEN-5: Arena allocator freed while child process still references `argv`/`envp` pointers

**Assumption that failed**: The arena allocator in `forkImpl` can be freed after `forkPty` returns because the parent doesn't need the strings anymore.

**File**: `zig/pty_unix.zig:49-51` and `zig/lib.zig:105-158`

**Concrete proof**: In `forkImpl`:
```zig
var arena = std.heap.ArenaAllocator.init(alloc);
defer arena.deinit();  // <-- frees all strings
const a = arena.allocator();
const file = try pty.getStringAlloc(env, argv[0], a);
// ... builds exec_argv, exec_envp using arena allocator ...
const result = lib.forkPty(.{
    .file = file.ptr,
    .argv = @ptrCast(exec_argv.ptr),
    .envp = @ptrCast(exec_envp.ptr),
    // ...
});
// arena.deinit() runs here — frees file, exec_argv, exec_envp
```

Inside `forkPtyUnix` (`lib.zig:105-158`):
- Line 118: `forkpty()` is called — this calls `fork()` internally
- Line 125: In the child process (pid == 0), the code calls `platform.execChild(opts.file, opts.argv, opts.envp)`
- Line 142-146: In the parent, `forkPtyUnix` returns, then `forkImpl` returns, then `arena.deinit()` frees all the strings

The key question: does the child process still reference the arena-allocated memory?

After `fork()`, the child has a copy-on-write copy of the parent's address space. The `opts.file`, `opts.argv`, `opts.envp` pointers point into the arena allocation in the child's (copied) address space. When `execChild` calls `execvpe`/`execvp`, the kernel reads these pointers from the child's address space.

**This is actually safe** because `fork()` gives the child its own address space, and `arena.deinit()` only runs in the parent process (after `forkPtyUnix` returns to the parent). The child calls `execvpe` which replaces the entire address space before the parent's `arena.deinit()` runs.

**HOWEVER**, on macOS specifically, the code path is:
```zig
pub fn execChild(file, argv, envp) void {
    const environ = @extern(..., .{ .name = "environ" });
    environ.* = envp;  // writes arena-allocated pointer to global
    _ = execvp(file, argv);
}
```

If `execvp` fails (e.g., file not found), the child calls `std.process.exit(1)` (lib.zig:143). During exit, any `atexit` handlers run, and they may access `environ` — which points to the arena-allocated `envp`. In the child process, the arena memory is still valid (CoW copy), so `atexit` handlers accessing `environ` will work. This is safe.

**The actual PROVEN bug is different**: the arena allocator in the NAPI layer (`pty_unix.zig:49-51`) allocates strings using `alloc` which is `std.heap.c_allocator`. The `ArenaAllocator` wraps `c_allocator`. After `forkPty` returns to the parent, `arena.deinit()` frees the backing memory using `c_allocator.free()` which calls `libc free()`. This is correct for the parent.

But: the exit monitor thread (`pty_unix.zig:16-31`) uses `alloc.create(lib.ExitInfo)` which uses `c_allocator`. The arena is unrelated to the exit monitor — this is fine.

**Re-examining**: The real issue is that the `forkImpl` function allocates `exec_argv` and `exec_envp` on the arena, then passes raw pointers (`.argv = @ptrCast(exec_argv.ptr)`) into `ForkOptions`. `forkPtyUnix` stores these in the `ForkOptions` struct on the stack. `forkpty()` internally calls `fork()`. The child inherits CoW memory. The child calls `execChild` with the pointers from `opts` (which are copies in the `ForkOptions` struct). The `ForkOptions` struct is on the parent's stack frame of `forkPtyUnix` — the child has a CoW copy of this stack. So the child's `opts.file`, `opts.argv`, `opts.envp` point to the child's CoW copy of the arena memory. `execvpe`/`execvp` reads these before the parent frees them. This is safe.

**CORRECTION**: After thorough analysis, this is actually **HARDENED** — fork creates CoW copies, and exec happens before the parent frees arena. Moving to HARDENED.

### PROVEN-5 (Replacement): `WriteQueue` silently drops data on non-EAGAIN write errors

**Assumption that failed**: `WriteQueue._process` handles all write errors appropriately.

**File**: `src/pty/_writeQueue.ts:44-50`
```typescript
if (err) {
    if ("code" in err && err.code === "EAGAIN") {
        this._immediate = setImmediate(() => this._process());
        return;
    }
    this._queue.length = 0;  // silently drops ALL queued data
    return;
}
```

**Concrete proof**: When `fs.write` returns a non-EAGAIN error (e.g., `EIO`, `ENOSPC`, `EPIPE`), the WriteQueue silently discards the entire queue including the current partially-written buffer and all subsequent pending writes. No error is propagated to the caller. No callback, no event, no exception.

Concrete scenario:
```typescript
const pty = spawn("/bin/sh");
// Fill the write queue with multiple messages
pty.write("command1\n");
pty.write("command2\n");
pty.write("command3\n");
// If command1's write gets EIO (e.g., child process crashes mid-write):
// - command1 is partially written or lost
// - command2 and command3 are silently dropped
// - No error reported to the user
// - pty.write() continues to accept data but it goes nowhere
```

The `enqueue` method returns the byte length as if the write succeeded (line 23: `return buf.length`), even though the data may be silently dropped later when `_process` encounters the error. There is no mechanism to report write failure back to the caller.

**Root cause / WHY**: Error handling was simplified for the common case (EAGAIN retry). Non-EAGAIN errors are considered fatal for the queue but no notification channel exists to inform the application layer.

**Meta-pattern**: "Silent data loss on error in async write queue — fire-and-forget write API with no error propagation."

### SUSPECTED-5: Windows `WinConPtyContext` use-after-free if `close()` and exit monitor race

**What was attempted**: In `zig/win/napi.zig:381-400`, `closeImpl` calls `win.killProcess(ctx.spawn_result.process, 1)` on the `WinConPtyContext`. Meanwhile, `winExitMonitorThread` (line 84-115) also accesses `ctx.spawn_result` fields. The exit monitor eventually calls `alloc.destroy(ctx)` (line 114).

If `closeImpl` fires from JS at the exact moment the exit monitor is in the middle of its cleanup sequence (e.g., after `waitForExit` returned but before `closeConin`), there's a potential race where both the JS thread and the exit monitor thread access `ctx.spawn_result` concurrently.

Specifically: `closeImpl` calls `killProcess(ctx.spawn_result.process, 1)`. If the exit monitor has already called `closePty(&ctx.spawn_result)` which sets `process = INVALID_HANDLE`, then `killProcess` receives `INVALID_HANDLE` — `TerminateProcess(INVALID_HANDLE, 1)` fails silently (returns 0). This is benign.

But if the exit monitor calls `alloc.destroy(ctx)` while `closeImpl` is still executing `win.killProcess(ctx.spawn_result.process, 1)`, the `ctx` pointer is freed and `ctx.spawn_result.process` is a use-after-free read.

**Why proof couldn't be constructed**: The timing window is extremely narrow — `killProcess` is a single `TerminateProcess` call which reads the handle once. The destroy happens after all cleanup including `napi_call_threadsafe_function` (blocking), which requires the JS thread. Since `closeImpl` runs on the JS thread, and the exit monitor's tsfn callback also runs on the JS thread, there's an implicit serialization. The destroy only happens after the tsfn callback completes on the JS thread, so `closeImpl` would have already returned. This likely makes the race impossible in practice.

**Runtime test**: Stress test calling `close()` immediately after spawn on Windows with fast-exiting processes.

### HARDENED-1: Signal masking around `forkpty()` prevents signal-delivery race

**File**: `zig/lib.zig:113-115`
```zig
var sigset_all = std.posix.sigfillset();
var sigset_old: std.posix.sigset_t = undefined;
```

**Attacks tried**:
1. Sending SIGCHLD to the parent during fork — blocked by sigfillset mask
2. Sending SIGTERM during fork — blocked, restored after fork

**Result**: The signal mask is properly saved, set to block all, and restored after fork in both parent (line 146) and child (line 126). Child also resets all signal handlers to default (via `resetSignalHandlers`). This defense matches the node-pty pattern and is correct.

**Dependencies**: Depends on `sigprocmask` being async-signal-safe (it is per POSIX). Depends on `forkpty` respecting the signal mask (it does — it calls `fork` internally).

### HARDENED-5: Arena allocator + fork CoW — child process has its own copy of allocated strings

**File**: `zig/pty_unix.zig:49-51` + `zig/lib.zig:118-143`

**Attacks tried**:
1. Parent frees arena while child is still using strings — child has CoW copy, safe
2. `execvpe` fails and child exits — `atexit` handlers may access arena memory, but it's the child's CoW copy
3. Concurrent spawn calls sharing arena — each `forkImpl` call has its own arena

**Result**: The arena allocator pattern is safe because `fork()` creates a copy-on-write snapshot of the address space. The child's copies of argv/envp are valid until `exec` replaces the address space. The parent's `arena.deinit()` only affects the parent's copy.

**Dependencies**: Depends on `fork()` providing CoW semantics (guaranteed by POSIX on all supported platforms).

## Root Causes

### Root Cause 1: Stream granularity assumptions
**Findings**: PROVEN-2 (flow control), SUSPECTED-6 (data loss on exit)
**Meta-pattern**: Code assumes data arrives in specific-sized chunks (single characters, complete messages). In reality, `tty.ReadStream` coalesces reads, and event loop scheduling determines delivery order.

### Root Cause 2: Missing cross-thread coordination in teardown
**Findings**: PROVEN-3 (close vs exit monitor race), SUSPECTED-4 (thread leak), SUSPECTED-5 (Windows use-after-free)
**Meta-pattern**: `close()` performs local cleanup but doesn't coordinate with background threads (exit monitor, read thread). Relies on SIGHUP/killProcess to eventually cause threads to terminate, with no join or synchronization.

### Root Cause 3: Promise lifecycle incomplete
**Findings**: PROVEN-4 (waitFor hangs on exit), SUSPECTED-6 (data lost before exit callback)
**Meta-pattern**: Promises created for data-dependent operations (waitFor) don't account for the data source terminating. No "process exited" rejection path.

## Unexplored (UNCLEAR)

### UNCLEAR-1: macOS `P_COMM_OFFSET = 243` hardcoded offset into `kinfo_proc`

**File**: `zig/pty_darwin.zig:42`

The offset into `kinfo_proc` for `p_comm` is hardcoded as 243. If Apple changes the struct layout in a future macOS version, this silently reads garbage. Deferred because verifying would require testing across macOS versions.

---

### UNCLEAR-2: Linux `getdents64` fallback — alignment and struct layout assumptions

**File**: `zig/pty_linux.zig:66-81`

The `getdents64` fallback path casts raw bytes to `*align(1) const linux.dirent64`. This depends on the `linux.dirent64` struct layout matching the kernel's. Deferred because it requires a kernel version where `close_range` fails but `/proc/self/fd` exists.

---

### UNCLEAR-3: Build system cleans `prebuilds/` on every build

**File**: `build.zig:44`
```zig
const clean = &b.addRemoveDirTree(.{ .cwd_relative = "prebuilds" }).step;
```

Every build step depends on `clean`, which removes the entire `prebuilds/` directory. If a user builds one target, all other prebuilt binaries are deleted. This is a design choice but could surprise users who expect incremental builds to preserve other targets.
